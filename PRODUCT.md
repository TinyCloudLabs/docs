# Product

## Register

brand

## Users

Application developers building browser or Node.js integrations, protocol implementers checking capability and delegation behavior, and operators deploying TinyCloud nodes or using the CLI. They arrive with a concrete job and need the shortest trustworthy route to a working result.

## Product Purpose

TinyCloud documentation teaches developers and operators how to build with, understand, and run a user-owned personal data cloud. Success means readers can select the right journey immediately, complete a verified first task, and find exact source-backed behavior without confusing tutorials, concepts, and reference material.

## Brand Personality

Sovereign, technical, and direct. The experience should create confidence through precise language, visible trust boundaries, and practical paths rather than hype.

## Anti-references

Do not resemble an uncustomized Mintlify starter kit, a generic SaaS documentation template, or a marketing page that hides operational constraints behind abstract claims. Avoid decorative complexity that competes with navigation and code.

## Design Principles

- Lead with reader intent: Build, Understand, and Operate should always be obvious entry points.
- Earn trust with exactness: distinguish verified behavior, boundaries, beta versions, and known limitations.
- Make progress visible: every tutorial should state prerequisites, actions, and a recognizable success result.
- Keep theory and action connected: concepts should clarify implementation decisions, not become detours.
- Make TinyCloud unmistakable: use TinyCloud identity and language instead of framework defaults.

## Accessibility & Inclusion

Meet WCAG 2.1 AA for text, controls, focus, and color contrast in light and dark modes. Preserve keyboard navigation, descriptive labels, reduced-motion preferences, and readable line lengths. Do not rely on color alone to communicate navigation state or meaning.
