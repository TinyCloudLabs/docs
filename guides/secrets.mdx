TinyCloud Secrets provides encrypted key-value storage with client-side encryption. The server never sees your plaintext data — all encryption and decryption happens on the client using keys derived from your wallet signature.

## Quick Start

<CodeGroup>
```typescript Node SDK
import { TinyCloudNode, PrivateKeySigner } from '@tinycloud/node-sdk';

const tc = new TinyCloudNode({
  host: 'https://node.tinycloud.xyz',
  privateKey: process.env.TC_PRIVATE_KEY,
  autoCreateSpace: true,
});
await tc.signIn();

// Unlock the vault (derives encryption keys from wallet signature)
const signer = new PrivateKeySigner(process.env.TC_PRIVATE_KEY);
await tc.vault.unlock(signer);

// Store an encrypted secret
await tc.vault.put('secrets/API_KEY', { value: 'sk-abc123' });

// Retrieve it
const result = await tc.vault.get('secrets/API_KEY');
if (result.ok) {
  console.log(result.data.value); // { value: 'sk-abc123' }
}
```

```bash CLI
# Set your private key
export TC_PRIVATE_KEY=your-64-char-hex-key

# Store a secret
tc secrets put API_KEY sk-abc123

# Retrieve it
tc secrets get API_KEY

# List all secrets
tc secrets list
```
</CodeGroup>

## How It Works

<Steps>
  <Step title="Unlock the Vault">
    The vault derives encryption keys from your wallet signature. This happens once per session — subsequent operations use the cached keys.
  </Step>
  <Step title="Store Encrypted Data">
    When you store a secret, the client generates a random data key, encrypts your value with AES-256-GCM, and encrypts the data key with your master key. Only encrypted blobs are sent to the server.
  </Step>
  <Step title="Retrieve and Decrypt">
    When you read a secret, the client fetches the encrypted blob and data key, decrypts the data key with your master key, then decrypts the value. Decryption is entirely client-side.
  </Step>
</Steps>

## Vault Operations

### Put

Store an encrypted value.

<CodeGroup>
```typescript Node SDK
const result = await tc.vault.put('secrets/DATABASE_URL', {
  value: 'postgres://user:pass@host/db',
  createdAt: new Date().toISOString(),
});

if (!result.ok) {
  console.error(result.error.message);
}
```

```bash CLI
# Inline value
tc secrets put DATABASE_URL "postgres://user:pass@host/db"

# From file
tc secrets put TLS_CERT --file ./cert.pem

# From stdin
echo "secret-value" | tc secrets put MY_SECRET --stdin
```
</CodeGroup>

### Get

Retrieve and decrypt a value.

<CodeGroup>
```typescript Node SDK
const result = await tc.vault.get('secrets/DATABASE_URL');

if (result.ok) {
  console.log(result.data.value);
}
```

```bash CLI
# JSON output
tc secrets get DATABASE_URL
# {"name":"DATABASE_URL","value":"postgres://..."}

# Raw value (for piping)
tc secrets get DATABASE_URL --raw
# postgres://user:pass@host/db

# Write to file
tc secrets get TLS_CERT --output ./cert.pem
```
</CodeGroup>

### List

List all secret names.

<CodeGroup>
```typescript Node SDK
const result = await tc.vault.list({ prefix: 'secrets/' });

if (result.ok) {
  console.log(result.data); // ['API_KEY', 'DATABASE_URL']
}
```

```bash CLI
tc secrets list
# {"secrets":["API_KEY","DATABASE_URL"],"count":2}
```
</CodeGroup>

### Delete

Delete a secret.

<CodeGroup>
```typescript Node SDK
const result = await tc.vault.delete('secrets/OLD_KEY');
```

```bash CLI
tc secrets delete OLD_KEY
```
</CodeGroup>

## Sharing Secrets

You can share vault-encrypted secrets with other users using vault grants and delegations. This uses X25519 Diffie-Hellman key exchange to re-encrypt the data key to the recipient's public key.

<Warning>
Both a **vault grant** and a **delegation** are required. The vault grant provides the decryption key; the delegation provides access to your space's KV data. Without both, the recipient cannot read the secret.
</Warning>

### Grant Access

<CodeGroup>
```typescript Node SDK
// Grant access to another user by their DID
const result = await tc.vault.grant(
  'secrets/SHARED_API_KEY',
  'did:pkh:eip155:1:0x70997970...'
);
```

```bash CLI
tc vault grant secrets/SHARED_API_KEY --to did:pkh:eip155:1:0x70997970...
```
</CodeGroup>

<Note>
The recipient must have unlocked their vault at least once to publish their X25519 public key. If you get a `PUBLIC_KEY_NOT_FOUND` error, ask the recipient to run `tc vault unlock` first.
</Note>

### Revoke Access

Revoking rotates the data key and re-grants to remaining authorized users.

<CodeGroup>
```typescript Node SDK
await tc.vault.revoke(
  'secrets/SHARED_API_KEY',
  'did:pkh:eip155:1:0x70997970...'
);
```

```bash CLI
tc vault revoke secrets/SHARED_API_KEY --from did:pkh:eip155:1:0x70997970...
```
</CodeGroup>

### Read a Shared Secret

The recipient uses the delegation token and `getShared` to decrypt.

<CodeGroup>
```typescript Node SDK
import { deserializeDelegation } from '@tinycloud/node-sdk';

// Apply the delegation (grants KV access to the grantor's space)
const delegation = deserializeDelegation(delegationJson);
const access = await tc.useDelegation(delegation);

// Fetch and decrypt the shared secret
const result = await tc.vault.getShared(
  'did:pkh:eip155:1:0xf39Fd6e5...', // grantor's DID
  'secrets/SHARED_API_KEY',
  { kv: access.kv }
);

if (result.ok) {
  console.log(result.data.value);
}
```

```bash CLI
tc vault get-shared \
  did:pkh:eip155:1:0xf39Fd6e5... \
  secrets/SHARED_API_KEY \
  --delegation-file delegation.json \
  --raw
```
</CodeGroup>

## Error Handling

All vault operations return a `Result` type:

```typescript
const result = await tc.vault.get('secrets/MY_KEY');

if (result.ok) {
  // Success: result.data contains the value
  console.log(result.data.value);
} else {
  // Error: result.error contains code and message
  switch (result.error.code) {
    case 'NOT_FOUND':
      console.log('Secret does not exist');
      break;
    case 'VAULT_LOCKED':
      console.log('Call vault.unlock() first');
      break;
    case 'DECRYPTION_FAILED':
      console.log('Key may have been rotated');
      break;
  }
}
```

| Error Code | Cause | Fix |
|-----------|-------|-----|
| `VAULT_LOCKED` | Vault not unlocked | Call `vault.unlock(signer)` |
| `NOT_FOUND` | Key doesn't exist | Check key name |
| `PUBLIC_KEY_NOT_FOUND` | Recipient hasn't unlocked vault | Recipient runs `vault unlock` |
| `GRANT_NOT_FOUND` | No vault grant for this user | Grantor runs `vault grant` |
| `DECRYPTION_FAILED` | Key was rotated after grant | Grantor re-runs `vault grant` |

## Security Model

- **Client-side encryption**: All cryptographic operations happen in the browser (WASM) or Node.js SDK. The server never sees plaintext.
- **Per-entry keys**: Each secret has its own randomly generated data key, encrypted with your master key.
- **Master key derivation**: Your master key is derived from a deterministic wallet signature using HKDF. It's never stored — it's re-derived each time you unlock.
- **X25519 sharing**: When sharing, the data key is re-encrypted to the recipient's public key via Diffie-Hellman. The recipient decrypts with their private key.
- **Key rotation on revoke**: Revoking a grant generates a new data key, re-encrypts the data, and re-grants to remaining authorized users.
